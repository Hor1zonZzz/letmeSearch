# AI 新闻 Topic 热度算法高层设计

## 目标

热度系统以 **Topic（真实事件）** 为排名单位，而不是直接给单条 Post 排名。多个账号对同一事件的原创、引用、回复或转发会先归入同一个 Topic，再汇总互动指标，从而避免同一事件重复占榜。

该阶段只使用确定性的 TypeScript 和 SQLite 计算，不依赖 Flue、Agent 或 LLM。
LLM 只负责前序的 Post 分类与 Topic 合并。

## 数据流

```text
Post → Topic → 指标快照 → 有效浏览量/增长速度 → 热度分数 → 热榜状态
```

1. 每 4 小时刷新仍在追踪的 Post 指标。
2. 根据最新快照计算各 Post 的浏览量增量和增长率。
3. 将同一 Topic 下的 Post 汇总为 Topic 指标。
4. 对所有候选 Topic 计算相对热度分数。
5. 根据晋级、上榜、下榜和停止追踪规则更新状态。

## Post 分类状态

前序分类产生三种决策：

- `important`：立即进入 Topic，并立即参与指标追踪。
- `observe`：立即进入 Topic 和指标追踪，但尚未认定为重要新闻。
- `ignore`：不进入 Topic，也不参与热度计算。

### Observe 晋级

`observe` Post 发布满 5 小时后，满足任一条件即晋级为 `important`：

- 累计浏览量达到 250,000；
- 相比首次指标快照新增至少 100,000 浏览量。

晋级只改变 Post 的编辑决策，不创建新 Topic。

## 指标快照

每次轮询为 Post 保存不可变快照，至少包含：

- 抓取时间；
- 浏览量；
- 点赞数；
- 转发数；
- 回复数；
- 引用数。

热度首版以浏览量和浏览增速为主。其他互动指标先保存，后续可以加入评分，而无需重新采集历史数据。

## Topic 有效浏览量

同一事件可能被多个账号重复传播，直接求和会高估事件影响力。因此采用折扣汇总：

```text
Topic 有效浏览量
= Topic 内最高单帖浏览量
+ 其他 Post 浏览量总和 × 30%
```

示例：同一 Topic 的三条 Post 分别有 800,000、300,000、100,000 浏览量：

```text
有效浏览量 = 800,000 + (300,000 + 100,000) × 30%
             = 920,000
```

这一规则保留多来源扩散信号，同时避免重复受众被完全累计。

## 上榜门槛

Topic 有效浏览量达到 **1,000,000** 后，才有资格进入热榜。

门槛与热度分数分离：

- 门槛判断 Topic 是否具备足够规模；
- 热度分数决定已达门槛 Topic 之间的相对排名。

## 热度分数

热度由规模和速度各占 50%：

```text
heat = viewScore × 50% + velocityScore × 50%
```

### 浏览规模分 `viewScore`

1. 对候选 Topic 的有效浏览量做对数变换，降低超大 Topic 的绝对数量优势；
2. 在当前候选集合内计算百分位；
3. 将结果归一化到 `0–1`。

### 增长速度分 `velocityScore`

1. 根据最近两个有效快照计算单位时间浏览增量；
2. 汇总为 Topic 的增长速度；
3. 做对数变换和候选集合内百分位归一化；
4. 将结果归一化到 `0–1`。

使用相对百分位而不是固定速度阈值，可以适应不同时段整体流量水平的变化。

## 下榜规则

已上榜 Topic 连续两次计算满足：

```text
heat < 0.35
```

则从热榜移除。要求连续两次是为了避免单次 API 波动或短时降速造成频繁上下榜。

下榜不等于删除：Topic、Post 和历史指标快照仍然保留。

## 停止追踪

满足任一条件后，停止继续刷新 Topic/Post 指标：

- 连续两次指标增长率低于 5%；
- Topic 中需要追踪的单帖均已超过发布后 72 小时。

停止追踪后保留最终排名状态及全部历史快照。若后续需要支持事件重新升温，应通过显式的重新激活规则实现，而不是自动恢复。

## 状态模型

建议将 Topic 的热度生命周期表达为：

```text
tracking → ranked → cooling → unranked
    └──────────────────────→ stopped
```

- `tracking`：持续采集指标，但尚未达到上榜门槛；
- `ranked`：已进入热榜；
- `cooling`：第一次低于下榜阈值，等待下一次确认；
- `unranked`：曾上榜但已连续两次低于阈值；
- `stopped`：已满足停止追踪条件。

实际存储可以使用较少的主状态字段，并用连续低热度次数、连续低增长次数等计数器表达中间状态。

## 调度与失败语义

- 指标任务每 4 小时运行一次；
- 单个 Post 指标请求失败不应阻止其他 Post 更新；
- 失败不写入伪造的零值快照，也不推进连续低增长计数；
- 排名只使用成功获取的最新快照；
- 同一轮计算应在事务内写入 Topic 汇总、热度分数和状态，避免部分更新。

## Phase 2 范围

首个实现阶段包括：

1. Post 指标快照表；
2. 每 4 小时的独立指标采集 CLI；
3. Observe 晋级；
4. Topic 有效浏览量与增长速度计算；
5. 热度百分位和热榜状态；
6. 下榜与停止追踪计数器。

暂不包括：个性化榜单、地域权重、互动质量评分、异常流量识别、人工置顶以及 Markdown 报告排序改造。
