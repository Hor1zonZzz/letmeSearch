① 先补 Context：把它想成学校报社

📋

Workflow = 班主任

决定先收消息、再筛选、再写稿、最后归档。它不负责发挥文采。

🤖

Agent = 记者的工作守则

规定模型是谁、写作原则是什么。本项目只有 一份记者守则。

🪑

Session = 独立课桌

同一个记者可以换课桌。每张桌子的上下文分开，不会把两个事件混在一起。

② Markdown 是谁写的？

一个“记者模板”，开出多个独立工作台。首轮按顺序执行，不是 6 个机器人同时乱跑。

Agent 实际交付什么？

不是直接随意输出整篇 Markdown，而是交付结构化草稿：

{ headline, summary, keyFacts, changeSummary }

谁负责最终排版？

确定性的 TypeScript “排版机”补上标题、更新时间、要点和可信来源链接，再写入 .md。

版本规则：同一事件来了新帖，但没有新事实，就不叫写稿 Agent；确有新事实，才新开 Session 生成 v2/v3。

本次真实结果

6 条重要帖得到 6 个不同指纹，所以是 6 个事件，没有发生合并。

一句话：现在能合并“身份字段基本相同”的帖子，不能保证合并“人看起来是同一件事”的帖子；事实去重目前也主要依赖标准化文字相同。

⑤ Twitter API：其实只有 1 种接口
